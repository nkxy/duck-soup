Field Restrictions

Team: 1 designer, 1 product manager, 3 devleopers, 1 QA

Role: Product Design

Duration: January - April 2020

Responsibilities: Interaction design, visual design, prototyping, usability testing, development handoff

The challenge

Veeva is a software company that focuses on pharmaceutical applications. OpenData is Veeva's database on health care providers and organizations, that pharmaceutical companies can purchase access to.

OpenData now would like to introduce premium data fields, which is additional data that customers can pay access to. As a data product, OpenData wants to be able to provide more data options for its customers.

OpenData Admins use Veeva Network, a data management tool, to configure OpenData. Network is a large, complex product used for various applications besides OpenData. The goal is to enable OpenData admins to set up and configure these premium fields inside Network for OpenData users.

Where should field restrictions go?

I first explored allowing OpenData Admins to specify certain fields as being restricted for specific users. There were many possible places in Network that we could build this feature into, all with different constraints. To explore further, I created sketches to quickly get my ideas down and discuss with the team.

The Data Visibility Profile (DVP) is assigned to a specific user, and contains country-specific information around permissions and layout.

Field restrictions are also country-specific, so combining the two could work. But this also reduces flexibility for the OpenData admin, since a new DVP needs to be made for every new field restriction, which could become very cumbersome.

To solve the flexibility issue, I tried putting the field restrictions next to where the DVP is assigned in the User Profile. However, OpenData admins would have to go configure each field one by one for each user, which is quite tedious.

Here OpenData admins would go to a new side tab, where they would be able to create a field restriction group. This would have a group of users, along with which fields are restricted for them.

This approach would be more flexible, as it would allow configuration of multiple fields for multiple users. However, this feature would be hidden underneath the Users tab, which could be hard to find.
